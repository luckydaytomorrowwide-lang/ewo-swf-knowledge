# ADR: 論理的に問題がない場合の再現試験優先運用（EWO再現試験）

- Status: Accepted
- Date: 2026-03-05
- Scope: EWO再現試験の運用（case-001 ExampleTest01 を起点とする）

## Context
同一の EWO / input / table を用いて再実行したにもかかわらず、取得された log の途中経過（SQL条件やヒット件数）が一致しないケースが発生した。  
人為的なオペレーションミス（別ログ混入、ファイル取り違え、命名混在など）が最も疑わしい一方、ログだけから人為ミスの有無を完全に断定することは難しい。  
このため、ロジック修正や詳細解析に入る前に、ログ取得の再現性を運用で担保する必要がある。

## Decision
ロジックやテーブル上の問題が見当たらない場合は、解析や修正を検討する前に、当該テストを「再現試験要」として同一資材での再実行を依頼する。  
再現試験で log に差分がないことを確認してから、より詳細な解析を行う。

## Policy / Operation
再現試験（TEST02以降）は以下を標準運用とする。

1. EWO / input / table を固定して再実行し、log を採取する（資材差分を作らない）。
2. testフォルダおよびファイル命名を統一し、Workflow名と揃える（混線防止）。
   - 例：
     - ExampleTest01EWO.json
     - ExampleTest01.input0X.json
     - ExampleTest01.table0X.json
     - ExampleTest01.log0X.json
3. log の冒頭にある Workflow / Input / corr（相関ID）が揃っていることを確認してから格納・コミットする。
4. 再現試験の合否判定を機械的に行う。
   - `Workflow completed` が出ていること
   - 期待するキー（例: `nameValue`）が出ていること
   - 重大な逸脱（例: `parentId: undefined` 等）が出た場合は「再現試験未達」として再実行する
5. 再現試験で log に差分がないことを確認できた時点で、必要に応じて詳細解析（ログの経路差分、AC単位の入力/出力、SQL条件の妥当性確認）へ進む。

## Consequences
- ロジック修正や過剰な解析を始める前に、ログの再現性（証跡の信頼性）を確保できる。
- 追加コストとして、再現試験の再実行が発生する可能性があるが、誤った原因追跡の手戻りを抑制できる。
